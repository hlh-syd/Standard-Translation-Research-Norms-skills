# Frontend Prompts

Reuse these prompts verbatim or with minimal project-specific adaptation. Replace the source-path placeholders before use.

## Prompt for `项目讲解`

```markdown
你将扮演一名资深前端工程师、科研可视化设计师和科普讲解策展人。请为我的科研代码项目创建一个面向非专业听众的交互式前端讲解网站。

项目背景：
我的课题代码位于：
上一级目录下script中

先学习代码总结出5个这个项目主题。受众是非专业人员，他们不了解多组学、生存分析、Cox 模型、外部验证、可解释性等概念。因此网站必须用极其详细、图文并茂、交互性强的方式讲解我的课题主题、代码结构、数据流、模型流程和结果含义。

目标交付：
请在下面文件夹下创建一个完整可运行的前端项目：
项目讲解

请不要修改源课题目录：
源课题目录只允许读取、分析、引用，不允许改动。
核心任务：
1. 先审查源课题代码结构，理解 脚本、data/results 的关系。
2. 创建一个前端讲解项目，用于汇报和教学。
3. 网站需要帮助非专业人员理解：
   - 这个课题研究什么问题
   - 为什么需要多组学
   - 什么是xxx（方向）
   - 什么是xxxx(研究主题）
   - 什么是 xxx(模型)
   - 为什么要做xx
   - xx(数据集) 扮演什么角色
   ...

前端技术建议：
优先使用 Vite + React + TypeScript。
可使用 Framer Motion / GSAP / Three.js / React Three Fiber / D3 / ECharts / Tailwind CSS 等工具。
如果当前环境已有其它前端技术栈，也可以选择更稳妥的方案，但必须说明选择理由。

视觉与交互要求：
1. 页面必须“动效十足”，不能是普通静态文档页。
2. 页面切换、章节切换、流程展开、模型解释、卡片浮现、数据流动画都要有高质量动效。
3. 请搜索 Figma Community 或可访问的高质量 Figma 设计资源，参考并使用不少于 6 个高质量 Figma 组件/设计模式/动效组件。
4. 对每个 Figma 组件/设计模式，请在项目中建立一个 `design_sources.md`，记录：
   - 组件名称
   - 来源链接
   - 用途
   - 在本站中对应的实现位置
   - 是否直接使用、改造使用、或仅作为设计参考
5. 如果无法联网或无法访问 Figma Community，不要伪造搜索结果；请写入 `design_sources.md` 说明原因，并使用本地可实现的高质量替代方案，但仍需实现不少于 6 个“组件级动效模块”。

必须包含的页面/模块：
1. 首页 Hero
   - 用一句非专业人员能懂的话解释课题。
   - 加入动态 3D DNA / 细胞 / 网络节点 / 数据流视觉。
   - 提供“开始讲解”入口。

2. 课题总览页
   - 用故事线解释：输入到研究结果。
   - 使用流程图展示 当前项目的研究路线 。
   - 每一步都要有通俗解释和专业补充。

3. 数据地图页
   - 讲清楚 数据集 的作用。
   - 用交互式卡片或地图式布局展示每个数据集的用途、优点、限制。

4. 融合页
   - 用可视化方式解释 各个数据集的类型与方法设计 各自是什么。
   - 使用交互式“多条数据河流汇入一个 研究结局 ”的动画。

5~9. 为各个研究路线的节点页

10. 代码导览页
   - 自动或半自动读取源项目中的 01-07 脚本名称，生成代码流程导览。
   - 每个脚本都要说明：
     - 输入是什么
     - 输出是什么
     - 在全链路中的位置
     - 给非专业人员的解释
     - 给专业人员的补充说明


图文与 3D 要求：
1. 至少实现 3 个原创或程序生成的可视化模型：
   - 数据流 3D/2.5D 模型
   - 动态模型
   - 网络模型
2. 至少实现 6 个图文讲解组件：
   - 概念解释卡
   - 流程时间线
   - 数据集卡片
   - 模型对比卡
   - 风险提示卡
   - 结果解释卡
3. 图文说明必须丰富，不要只写标题。每个核心概念至少包含：
   - 一句话解释
   - 类比解释
   - 在本项目中的作用
   - 常见误解
   - 专业补充

工程要求：
1. 创建完整前端项目结构。
2. 提供 `README.md`，说明如何安装、运行、构建。
3. 提供 `design_sources.md`，记录 Figma/设计来源。
4. 提供 `content_map.md`，说明每个页面讲解了什么。
5. 所有页面必须可运行，不允许只有占位符。
6. 不要破坏源课题目录。
7. 如果读取源项目文件失败，要给出清楚 warning，不要静默跳过。
8. 如果某些数据或结果文件不存在，要在页面中以“数据缺失/待补充”的方式诚实展示，不要伪造结果。

验收标准：
1. 目标目录中存在完整前端项目。
2. `npm install` 和 `npm run dev` 或等价命令可运行。
3. 至少包含 10 个讲解页面/章节。
4. 至少包含 6 个组件级动效模块。
5. 至少包含 3 个 3D 或动态图形辅助解释。
6. 至少包含 `README.md`、`design_sources.md`、`content_map.md`。
7. 页面文字足够详细，适合非专业人员学习。
8. 对项目限制和降级规则有明确展示。
9. 不修改源课题代码目录。
10. 最后请运行构建或类型检查，并报告验证结果。

执行方式：
请直接创建项目并实现，不要只给方案。
如果发现关键信息不足，最多提出 1-2 个必要问题；若可以合理假设，请继续执行并在最终说明中列出假设。
最终回复请包含：
- 创建了哪些内容
- 如何运行
- 关键页面/组件清单
- 使用或参考的 Figma/设计来源
- 验证命令与结果
- 仍需人工补充或确认的地方

去 C:/.codex/skills下使用这些skills来辅助
UI/UX Intelligence Expert
frontend-skill
algorithmic-art
chart-visualization
figma
```
